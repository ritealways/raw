1. Business Objective & Problem Being Solved

Business problem: Health insurance members sometimes visit doctors outside the insurer's network (OON). Because there's no negotiated discount, the insurer pays more. Rising OON usage in a region also signals a network adequacy gap — not enough in-network doctors of a given specialty.

Why it matters: Today the network team learns about OON leakage from a quarterly report — i.e., after the money is already lost. By then a region may have leaked millions and members are frustrated.

Objective (SMART)

Forecast 1–6 months ahead the OON cost for each Region × Specialty combination, so the network team can proactively recruit doctors / renegotiate before costs spike.

Success metrics (business KPIs):
  • ↓ OON cost leakage by 8–12% in flagged regions within 2 quarters
  • Forecast accuracy: MAPE < 15% at the region-specialty level
  • Lead time: give the team ≥ 60 days to act
  • Precision on "rising cost" alerts > 70% (avoid crying wolf)

2. Why This Approach (Forecasting + Tree Models)

Option consideredVerdictReasoning
Classical ARIMA/SARIMA per series❌ Baseline onlyWe have hundreds of Region×Specialty series. One ARIMA per series = unmaintainable, and it can't use cross-features (network density, referrals).
Deep learning (LSTM/N-BEATS)❌ OverkillMonthly data → short histories (36–60 points). DL overfits, hard to explain to network business team.
Global tree model (XGBoost / RF)✅ ChosenOne model learns across ALL series, handles non-linear effects, mixes lags + business features, robust to outliers, and gives feature importance the business trusts.
Why tree-based (final): (1) Handles mixed features (numeric lags + categorical region/specialty), (2) No stationarity requirement, (3) Captures interactions (e.g., "cardiology + rural + low network density"), (4) Fast to retrain monthly, (5) SHAP explainability satisfies regulators/business.
Reframing: We turn forecasting into a supervised regression problem — predict next-month OON cost using lagged/rolling features. This is the modern "global forecasting" approach used in industry (M5 competition winners used LightGBM/XGBoost).

4. Target Variable

Target: oon_cost = total monthly out-of-network allowed_amount for each Region×Specialty, forecast for horizons h = 1..6 months.

We build a direct multi-horizon setup: separate target columns oon_cost_t+1 ... oon_cost_t+6, one model per horizon (or a single model with horizon as feature). Direct forecasting avoids compounding recursive errors.

Why forecast $ (not the OON rate)?

The network team's budget is in dollars. But we also track OON rate = OON claims / total claims as a secondary signal to distinguish "more members" from "genuine network gap."

Log-transform decision: OON cost is right-skewed and multiplicative in nature. We model log1p(oon_cost) to stabilize variance, then expm1() the prediction. This alone often drops MAPE by 2–4%.

8. Feature Creation & Feature Engineering

This is where a global tree model wins — we inject time structure via engineered features.

8.1 Lag features (memory of the series)

for lag in [1, 2, 3, 6, 12]:
    df[f'lag_{lag}'] = df.groupby(['region','specialty'])['y'].shift(lag)
Why these lags: lag_1/2/3 = short-term momentum; lag_6 = half-year; lag_12 = same month last year (seasonality). Chosen from ACF/PACF spikes at lag 1 and 12.

8.2 Rolling window features (trend & volatility)

g = df.groupby(['region','specialty'])['y']
df['roll_mean_3']  = g.shift(1).rolling(3).mean()
df['roll_mean_6']  = g.shift(1).rolling(6).mean()
df['roll_std_3']   = g.shift(1).rolling(3).std()
df['roll_max_6']   = g.shift(1).rolling(6).max()
df['mom_change']   = df['lag_1'] - df['lag_2']      # month-over-month momentum
df['trend_ratio']  = df['roll_mean_3'] / df['roll_mean_6']  # short vs long trend
Critical: .shift(1) before rolling so the window uses ONLY past data — prevents data leakage. roll_std_3 captures volatility (unstable series = network in flux). trend_ratio > 1 flags acceleration = the early-warning signal we want.

9. Feature Selection

  1. Drop leakage/constant: remove features derived from future, and zero-variance columns.
  2. Correlation filter: drop one of any pair with |corr| > 0.95 (e.g., roll_mean_3 vs roll_mean_2).
  3. Model-based importance: train XGBoost, rank by gain.
  4. Permutation importance on validation → keep only features that actually help unseen data.
  5. SHAP to confirm business logic (network density & lag_1 should top the list).

10. Hyperparameter Tuning

Why TimeSeriesSplit (not random CV): Random shuffling leaks future into past. We use expanding-window walk-forward validation to mimic real deployment.
Chart F — Walk-Forward (Expanding Window) CV

13. Production Steps (MLOps)

  1. Data pipeline: Airflow DAG runs monthly → pulls claims, aggregates, builds features (same code as training via shared module → no train/serve skew).
  2. Feature store: lag/rolling features cached (Feast) for reuse & consistency.
  3. Model registry: MLflow logs params, metrics, artifacts; versioned models.
  4. Retraining: automated monthly retrain; auto-promote only if new MAPE ≤ old + tolerance (champion/challenger).
  5. Batch inference: generate all Region×Specialty forecasts, write to a warehouse table.
  6. Serving: results piped to a Power BI / Tableau dashboard for the network team.
  7. Monitoring: track live MAPE vs forecast, data drift (PSI on feature distributions), and volume anomalies → alert on Slack.
  8. Fallback: if model fails, serve Seasonal Naïve so business never gets a blank report.

16. Interview Questions & Answers

Q1. Why did you use tree models instead of ARIMA/LSTM for time series?

We have hundreds of short monthly series. ARIMA needs one model per series (unscalable) and can't ingest business features like network density. LSTM overfits short histories and lacks explainability. A global XGBoost learns across all series, uses lags + domain features, handles non-linearity, and gives SHAP explanations the network team trusts. It also won the M5 forecasting competition benchmark.

Q2. Trees can't extrapolate a trend — how did you handle rising OON cost?

True — a tree can't predict a value higher than any it saw. I mitigate this by (1) modeling log1p and using lag/rolling-mean features so the model predicts on the recent level, (2) adding momentum features (mom_change, trend_ratio) so it captures acceleration, and (3) optionally modeling the difference/growth rate instead of raw level so extrapolation happens outside the tree.

Q3. How did you prevent data leakage?

Four guards: (1) .shift(1) before every rolling window so features use only past data; (2) all imputation/percentile stats fit on train only; (3) TimeSeriesSplit walk-forward CV, never random shuffle; (4) target encoding computed inside CV folds.

Q4. Why both ADF and KPSS tests?

They have opposite null hypotheses (ADF H0 = non-stationary; KPSS H0 = stationary). Using both gives a robust conclusion. If ADF says non-stationary AND KPSS rejects stationary, we're confident it's non-stationary and needs differencing for the ARIMA baseline.

Q5. If trees don't need stationarity, why test it at all?

Stationarity isn't required for trees, but the tests inform (a) the design of ARIMA baselines and (b) my understanding of trend/seasonality, which drives which lag features to engineer (e.g., lag_12 from the seasonal signal).

Q6. How did you choose your lag features?

From ACF/PACF: PACF cut off after lag 1–2 (short memory) and ACF showed a spike at lag 12 (yearly seasonality). So I used lags 1,2,3,6,12 — short momentum plus seasonal.

Q7. Why MAPE, and what's its weakness?

MAPE is % based so it's comparable across differently-sized regions and business-friendly. Weakness: it explodes when actuals are near zero and penalizes over-forecasts asymmetrically. So I also report WAPE (weighted) and RMSE for large-miss sensitivity.

Q8. Direct vs recursive multi-step forecasting — which and why?

I used direct (separate target per horizon or horizon as a feature). Direct avoids compounding errors of recursive forecasting where predicted values feed back as inputs. Trade-off: more models, but at monthly grain that's cheap.

Q9. How do you generate prediction intervals with XGBoost?

Train quantile models with the pinball (quantile) loss at q=0.1 and q=0.9 to get an 80% interval. This gives the network team confidence bounds so they act only on high-certainty rising alerts.

Q10. How would you handle a brand-new Region×Specialty with no history (cold start)?

Fall back to the hierarchy: use the parent region's or specialty's average behavior, or a similar segment's model, until enough history accrues. The global model already generalizes across segments, which helps cold start more than per-series ARIMA would.

Q11. How do you know the model is actually creating business value?

A/B / before-after: track actual OON cost in flagged regions where the network team acted vs a control group. Success = lower realized OON growth + high directional-alert precision. We report "$ prevented," not just MAPE.

Q12. How do you monitor the model in production?

Monthly: compute realized MAPE vs forecast, run PSI drift checks on feature distributions, alert if drift > 0.2 or MAPE degrades beyond tolerance. Champion/challenger auto-retrain and only promote if the new model wins. Seasonal-Naïve fallback if pipeline fails.

Q13. Why log-transform the target?

OON cost is right-skewed and multiplicative. log1p stabilizes variance so the model isn't dominated by huge segments, and it makes errors proportional — appropriate for cost data. I inverse with expm1 and apply a small bias correction.

Q14. RF vs XGBoost — why XGBoost as final?

RF is robust with little tuning (good fallback), but XGBoost's boosting captured the momentum/trend interactions better, giving ~1.6 pts lower MAPE. XGBoost also supports quantile loss for intervals and has efficient handling of our features. I kept RF as a stable fallback.

Q15. What are the biggest risks/limitations of this project?

(1) Trend extrapolation limits of trees; (2) regulatory/policy shocks (e.g., a new law) that no historical feature captures; (3) sparse segments; (4) feedback effect — once the team acts on alerts, the target changes, so we must retrain and evaluate carefully to attribute impact.

✅ Summary

We reframed OON cost forecasting as a global supervised regression using engineered lag + rolling + domain features, benchmarked against Naïve/SARIMA baselines, and finalized XGBoost (RF fallback) with walk-forward CV and Optuna tuning. It forecasts 1–6 months ahead per Region×Specialty with prediction intervals, feeding an early-warning dashboard so the network team recruits doctors before costs spike — shifting the business from reactive quarterly reports to proactive network management.
